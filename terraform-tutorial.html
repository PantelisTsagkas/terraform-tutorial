What is Infrastructure as Code — and why Terraform?

Infrastructure as Code (IaC)

Traditionally, cloud resources (servers, databases, networks) were created by clicking through a web console or running ad-hoc scripts. Infrastructure as Code means expressing that infrastructure in declarative configuration files you can version, review, and reuse — just like application code.

📜

Version Control

Track every infrastructure change in Git. Roll back to any previous state.

🔁

Repeatability

Deploy identical environments for dev, staging, and prod with one command.

🤝

Collaboration

Review infra changes via pull requests before anything touches production.

🧪

Testing

Validate configurations before applying. Catch drift automatically.

Where does Terraform fit?

Terraform is an open-source IaC tool by HashiCorp. You describe what you want (a VPC, an EC2 instance, a DNS record), and Terraform figures out how to create, update, or delete it. It's cloud-agnostic — the same workflow works across AWS, Azure, GCP, and hundreds of other providers.

Terraform vs. alternatives

ToolApproachBest for
TerraformDeclarative HCL, multi-cloudMulti-cloud, team workflows, state management
CloudFormationDeclarative YAML/JSON, AWS onlyAWS-only shops, deep AWS integration
PulumiImperative, real languages (Python/TS)Developers who prefer code over DSL
AnsibleProcedural, config managementServer config, patching (not great for cloud provisioning)
💡
Key Terraform concept: Declarative vs Imperative

You don't write "create this server, then that network" — you write "the desired end state". Terraform determines the operations needed to reach it.

Installation & Setup

Terraform ships as a single binary — no dependencies, no runtimes. Pick your OS below.

The easiest method on macOS is Homebrew:

Terminal bash
# Install via Homebrew (recommended)
brew tap hashicorp/tap
brew install hashicorp/tap/terraform

# Verify installation
terraform -version
# → Terraform v1.9.x

On Ubuntu/Debian, use the official HashiCorp apt repository:

Terminal bash
# Add the HashiCorp GPG key and repo
wget -O- https://apt.releases.hashicorp.com/gpg | \
  sudo gpg --dearmor -o /usr/share/keyrings/hashicorp-archive-keyring.gpg

echo "deb [signed-by=/usr/share/keyrings/hashicorp-archive-keyring.gpg] \
  https://apt.releases.hashicorp.com $(lsb_release -cs) main" | \
  sudo tee /etc/apt/sources.list.d/hashicorp.list

sudo apt update && sudo apt install terraform

# Verify
terraform -version

Use Chocolatey or download the binary directly from HashiCorp:

PowerShell (Admin) powershell
# Install via Chocolatey
choco install terraform

# Or via winget
winget install --id Hashicorp.Terraform

# Verify in a new terminal
terraform -version

Enable tab completion

Terminal bash
# For bash
terraform -install-autocomplete

# Reload your shell
source ~/.bashrc

Recommended editor setup

Install VS Code with the HashiCorp Terraform extension. It gives you syntax highlighting, autocomplete, and inline docs.

03

Providers, Resources & Configuration Files

The HCL language

Terraform configuration is written in HCL (HashiCorp Configuration Language) — a human-readable format that lives in .tf files. Every Terraform project is a directory of .tf files that Terraform treats as one combined configuration.

Providers

A provider is a plugin that lets Terraform talk to a specific API — AWS, Azure, GCP, GitHub, Cloudflare, Kubernetes, and hundreds more. You declare which providers you need in a required_providers block:

main.tf — Provider configuration hcl
terraform {
  required_version = ">= 1.9"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"   # ~> means "compatible with 5.x"
    }
  }
}

# Configure the provider (credentials, region, etc.)
provider "aws" {
  region = "eu-west-1"
}
🔑
AWS credentials

Never hard-code AWS keys in .tf files. Use the AWS CLI (aws configure), environment variables (AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY), or an IAM role instead.

Resources

A resource is any piece of infrastructure you want Terraform to manage. The syntax is always: resource "<TYPE>" "<LOCAL_NAME>" { ... }

main.tf — AWS S3 bucket example hcl
# Create an S3 bucket
resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-unique-demo-bucket"

  tags = {
    Environment = "dev"
    Team        = "platform"
  }
}

# Enable versioning on the bucket (separate resource)
resource "aws_s3_bucket_versioning" "versioning" {
  bucket = aws_s3_bucket.my_bucket.id   # Reference the bucket above

  versioning_configuration {
    status = "Enabled"
  }
}

Referencing resources

Use <RESOURCE_TYPE>.<LOCAL_NAME>.<ATTRIBUTE> to reference one resource from another. This creates an implicit dependency — Terraform builds a dependency graph and creates resources in the right order automatically.

Data sources

Data sources let you read existing infrastructure that you didn't create with Terraform:

Reading an existing VPC hcl
# Look up an existing VPC by tag
data "aws_vpc" "existing" {
  tags = {
    Name = "production-vpc"
  }
}

# Use its ID in another resource
resource "aws_subnet" "new_subnet" {
  vpc_id     = data.aws_vpc.existing.id
  cidr_block = "10.0.100.0/24"
}

Typical project file structure

Project layout tree
my-terraform-project/
├── main.tf          # Core resources
├── variables.tf     # Input variable declarations
├── outputs.tf       # Output value declarations
├── providers.tf     # Provider configuration
├── versions.tf      # Terraform & provider version constraints
├── terraform.tfvars # Variable values (DO NOT commit secrets)
└── .terraform/      # Downloaded providers (git-ignored)

Variables, Outputs & State Files

Input Variables

Variables make your configuration reusable. Declare them in variables.tf and set values in terraform.tfvars:

variables.tf hcl
variable "environment" {
  description = "Deployment environment (dev/staging/prod)"
  type        = string
  default     = "dev"

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "Must be dev, staging, or prod."
  }
}

variable "instance_count" {
  description = "Number of EC2 instances"
  type        = number
  default     = 1
}

variable "allowed_cidrs" {
  description = "List of CIDRs to allow inbound traffic"
  type        = list(string)
  default     = ["0.0.0.0/0"]   # ⚠ exposes resource to entire internet — restrict in production
}
terraform.tfvars — Set values hcl
environment    = "staging"
instance_count = 3
allowed_cidrs  = ["10.0.0.0/8", "192.168.0.0/16"]

Reference variables in your config with var.<name>:

Using variables in main.tf hcl
resource "aws_instance" "app" {
  count         = var.instance_count
  instance_type = "t3.micro"
  ami           = data.aws_ami.latest.id

  tags = {
    Name = "app-${var.environment}-${count.index}"
  }
}

Output Values

Outputs expose values after a terraform apply — useful for passing data between modules or displaying connection details:

outputs.tf hcl
output "instance_public_ips" {
  description = "Public IP addresses of all instances"
  value       = aws_instance.app[*].public_ip
}

output "bucket_arn" {
  description = "ARN of the S3 bucket"
  value       = aws_s3_bucket.my_bucket.arn
}

output "db_password" {
  description = "Database password"
  value       = random_password.db.result
  sensitive   = true   # ← redacted from CLI output
}

State Files

Terraform records what it created in a state file (terraform.tfstate). This is how it knows what exists, what's changed, and what needs to be destroyed.

⚠️
Never manually edit terraform.tfstate

Use terraform state mv, terraform state rm, or terraform import for state manipulation. Editing the JSON file by hand can corrupt your state and leave resources orphaned.

🔒
State contains secrets

Database passwords and other sensitive values are stored in plaintext in the state file. Never commit terraform.tfstate to version control — use a remote backend (see Section 07).

Useful state commands

CommandDescription
terraform state listList all resources in state
terraform state show <resource>Show full details of a resource
terraform state mv A BRename/move resource in state
terraform state rm <resource>Remove resource from state (doesn't destroy it)
terraform import <resource> <id>Import an existing cloud resource into state

Modules & Reusability

A module is a directory of .tf files treated as a reusable unit. Every Terraform project is already a "root module". You create child modules to avoid repeating yourself.

Creating a module

modules/s3-website/main.tf hcl
# A reusable module that creates an S3 static website
resource "aws_s3_bucket" "website" {
  bucket = var.bucket_name
}

resource "aws_s3_bucket_website_configuration" "config" {
  bucket = aws_s3_bucket.website.id

  index_document { suffix = "index.html" }
  error_document { key    = "404.html"    }
}

output "website_url" {
  value = aws_s3_bucket_website_configuration.config.website_endpoint
}
modules/s3-website/variables.tf hcl
variable "bucket_name" {
  type        = string
  description = "Unique name for the S3 bucket"
}

Calling a module

root main.tf — Using the module hcl
module "marketing_site" {
  source      = "./modules/s3-website"   # local path
  bucket_name = "my-marketing-site-2024"
}

module "blog" {
  source      = "./modules/s3-website"
  bucket_name = "my-blog-site-2024"
}

# Use module outputs
output "marketing_url" {
  value = module.marketing_site.website_url
}

Public modules from the Terraform Registry

The Terraform Registry hosts thousands of community and verified modules. Use them to avoid re-inventing the wheel:

Using a Registry module hcl
# Use the official AWS VPC module
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.5.1"

  name = "my-vpc"
  cidr = "10.0.0.0/16"

  azs             = ["eu-west-1a", "eu-west-1b"]
  private_subnets = ["10.0.1.0/24", "10.0.2.0/24"]
  public_subnets  = ["10.0.101.0/24", "10.0.102.0/24"]

  enable_nat_gateway = true
}
✅
Module design principles

Keep modules focused on one responsibility. Avoid modules that do too much. Always pin the version when using registry modules to avoid surprise breaking changes.

The Terraform CLI Workflow

The core Terraform lifecycle involves five commands. Learn these cold — you'll run them hundreds of times.

initDownload providers & modules
validateCheck HCL syntax
planPreview changes
applyExecute changes
destroyRemove resources

terraform init

Run this once when starting a project, or after adding a new provider or module. It downloads plugins into .terraform/.

Terminal bash
terraform init

# Upgrade providers to latest allowed versions
terraform init -upgrade

# Reconfigure backend
terraform init -reconfigure

terraform plan

Shows exactly what Terraform would do without changing anything. Always read the plan before applying. + create ~ modify - destroy

Terminal bash
# Interactive plan
terraform plan

# Save plan to file (required for automation / CI)
terraform plan -out=tfplan

# Plan for a specific var file
terraform plan -var-file="prod.tfvars"

# Target a specific resource only
terraform plan -target=aws_instance.app

terraform apply

Terminal bash
# Interactive apply (asks "yes/no")
terraform apply

# Apply a saved plan (no prompt — safe for CI)
terraform apply tfplan

# Auto-approve (skip the prompt — use carefully!)
terraform apply -auto-approve
⚡
Always use a saved plan in CI/CD

Run plan -out=tfplan in one step, review it, then apply tfplan in the next. This guarantees what you reviewed is exactly what gets applied — no surprise changes between plan and apply.

terraform destroy

Terminal bash
# Destroy ALL resources in state (interactive prompt)
terraform destroy

# Destroy only a specific resource
terraform destroy -target=aws_instance.app

Other useful commands

CommandDescription
terraform fmtAuto-format all .tf files to canonical style
terraform validateValidates configuration syntax and internal consistency (does not access remote state or real cloud)
terraform outputPrint all output values from the last apply
terraform consoleInteractive expression evaluator (great for debugging)
terraform apply -replace=RESOURCEForce recreation of a specific resource (replaces the deprecated terraform taint)
terraform apply -refresh-onlySync state with real infrastructure without making changes (replaces the obsolete terraform refresh)
terraform graphOutput dependency graph in DOT format
⚠️
Use -target with extreme caution

The -target flag (for both plan and destroy) is intended for exceptional recovery scenarios only — not normal workflows. Targeting individual resources can produce partial or inconsistent infrastructure state where Terraform's dependency graph is no longer accurate. Use it to get out of a broken situation, then immediately run a full plan to verify everything is consistent.

✦ Exercise 03

Full lifecycle on a local resource

Use the local provider (no cloud account needed!) to practise the full workflow:

  1. Add hashicorp/local to your required_providers
  2. Create a local_file resource that writes "Hello Terraform!" to hello.txt
  3. Run init → plan → apply. Verify hello.txt was created.
  4. Change the content to "Hello World!" and run plan again. Notice the diff.
  5. Run destroy and verify the file is removed.
07

Remote State & State Backends

Local state files are fine for learning, but in a team or CI/CD environment you need remote state: a shared, encrypted, locked state file.

Why remote state?

👥

Team access

Everyone on the team works from the same state, not a local copy.

🔒

Encryption

State is stored encrypted at rest, not as a plaintext local file.

🚦

State locking

Prevents two people running apply simultaneously and corrupting state.

📜

Versioning

S3 versioning lets you roll back to any previous state.

S3 backend (AWS)

S3 stores the state file. For locking, you have two options depending on your Terraform version:

Option A — Native S3 lockfile (Terraform v1.10+, recommended for new projects): Terraform writes a .tflock file directly into S3. No extra infrastructure needed.

backend.tf — Native S3 locking (v1.10+) hcl
terraform {
  backend "s3" {
    bucket       = "my-company-terraform-state"
    key          = "production/vpc/terraform.tfstate"
    region       = "eu-west-1"
    encrypt      = true
    use_lockfile = true   # native S3 locking — no DynamoDB needed
  }
}

Option B — DynamoDB locking (legacy, still widely used in existing infrastructure): Historically, Terraform required a separate DynamoDB table for state locking. You'll encounter this pattern in many existing setups, so it's worth knowing.

backend.tf — DynamoDB locking (legacy / existing infra) hcl
terraform {
  backend "s3" {
    bucket         = "my-company-terraform-state"
    key            = "production/vpc/terraform.tfstate"
    region         = "eu-west-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"   # separate DynamoDB table
  }
}

Creating the backend infrastructure

bootstrap/main.tf — Create the S3 bucket & DynamoDB table hcl
# S3 bucket for state
resource "aws_s3_bucket" "state" {
  bucket = "my-company-terraform-state"
}

resource "aws_s3_bucket_versioning" "state" {
  bucket = aws_s3_bucket.state.id
  versioning_configuration { status = "Enabled" }
}

resource "aws_s3_bucket_server_side_encryption_configuration" "state" {
  bucket = aws_s3_bucket.state.id
  rule {
    apply_server_side_encryption_by_default {
      sse_algorithm = "aws:kms"
    }
  }
}

# DynamoDB table for state locking
resource "aws_dynamodb_table" "lock" {
  name         = "terraform-state-lock"
  billing_mode = "PAY_PER_REQUEST"
  hash_key     = "LockID"

  attribute {
    name = "LockID"
    type = "S"
  }
}

Remote state data source

Read outputs from another Terraform workspace's state file:

Reading another workspace's state hcl
data "terraform_remote_state" "vpc" {
  backend = "s3"
  config = {
    bucket = "my-company-terraform-state"
    key    = "production/vpc/terraform.tfstate"
    region = "eu-west-1"
  }
}

# Now use the VPC outputs from that state
resource "aws_instance" "app" {
  subnet_id = data.terraform_remote_state.vpc.outputs.private_subnet_id
  # ...
}
🏢
Terraform Cloud / HCP Terraform

HashiCorp's hosted platform provides remote state, run history, team access controls, and SSO. HCP Terraform offers a free tier suitable for small teams and learning environments. Configure with backend "remote" { hostname = "app.terraform.io" ... }.

Best Practices: Security, Version Control & Collaboration

Security

  • Never store secrets in .tf files. Use environment variables, AWS Secrets Manager, HashiCorp Vault, or sensitive = true variables fed via CI secrets.
  • Enable state encryption at rest (S3 SSE-KMS) and in transit (TLS).
  • Apply least privilege to the IAM role/user that runs Terraform. Separate roles for plan (read) and apply (write).
  • Use terraform validate + tfsec in CI to catch misconfigurations early.
.gitignore for Terraform projects gitignore
# Local state — NEVER commit
*.tfstate
*.tfstate.backup
.terraform/

# Secret-containing variable files
*.auto.tfvars
secrets.tfvars

# Plan output files
tfplan

# Override files
override.tf
override.tf.json
📌
DO commit .terraform.lock.hcl

This file is generated by terraform init and pins exact provider versions and their checksums — protecting against supply-chain drift. All team members and CI/CD pipelines get identical provider binaries. Treat it like package-lock.json: commit it, don't ignore it.

💡
Which .tfvars files to commit?

It depends on what's in them. Non-secret config (region, instance sizes, feature flags) is safe to commit — e.g. dev.tfvars, staging.tfvars. Files containing passwords, API keys, or tokens should never be committed. If you'd be uncomfortable with it in a public repo, keep it out.

Structuring for multiple environments

Two common patterns:

Workspaces approach bash
# Create and switch workspaces
terraform workspace new staging
terraform workspace new production
terraform workspace select staging

# Reference workspace in config
# locals { env = terraform.workspace }

Best for small projects with near-identical environments. State is isolated per workspace.

Directory structure tree
infra/
├── modules/
│   ├── vpc/
│   └── app-cluster/
├── envs/
│   ├── dev/
│   │   ├── main.tf

Preferred for production setups. Each environment has its own state and can diverge safely.

CI/CD Pipeline pattern

GitHub Actions — Terraform workflow yaml
name: Terraform

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  terraform:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: hashicorp/setup-terraform@v3

      - name: Init
        run: terraform init

      - name: Validate
        run: terraform validate

      - name: Plan
        run: terraform plan -out=tfplan
        env:
          AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
          AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}

      - name: Apply  # only on merge to main
        if: github.ref == 'refs/heads/main'
        run: terraform apply tfplan

Collaboration checklist

  • ✅ Remote backend with state locking enabled
  • ✅ .terraform.lock.hcl committed to Git
  • ✅ Provider and module versions pinned
  • ✅ terraform fmt enforced in CI (use fmt -check)
  • ✅ Plan output posted as PR comment (use atlantis or GitHub Actions)
  • ✅ tfsec or checkov security scanning in CI
  • ✅ Secrets via environment variables or Vault, never in .tf

Troubleshooting Common Errors

Here are the errors you'll encounter most often, with their causes and fixes.

Error: No valid credential sources found for AWS Provider
Fix: Run aws configure to set up credentials, or export AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY as environment variables. For EC2/Lambda, use an IAM instance role.
Error: Error locking state: Error acquiring the state lock
Fix: Another process holds the lock (or a previous run crashed). Check DynamoDB for a stale lock entry. If safe, run terraform force-unlock <LOCK_ID>. Never force-unlock if another apply might be running.
Error: Resource already exists / BucketAlreadyExists
Fix: The resource exists in the cloud but not in Terraform's state. Import it: terraform import aws_s3_bucket.my_bucket my-bucket-name, then rerun plan.
Error: Cycle in dependency graph
Fix: Two resources depend on each other, creating a circular dependency. Use depends_on carefully, or restructure so only one resource references the other.
Error: Provider configuration not present
Fix: You declared a resource but haven't added or configured the required provider block. Run terraform init after adding it.
Error: Reference to undeclared resource / module
Fix: You referenced aws_instance.web.id but the resource is named aws_instance.app. Check spelling of resource type and local name.
Plan shows unexpected destroy
Fix: Usually caused by renaming a resource (Terraform sees it as delete + create) or an attribute change that forces replacement. Use terraform state mv old_name new_name for renames, or lifecycle { create_before_destroy = true } to minimise downtime.

Debug techniques

Enabling verbose logging bash
# Set log level (TRACE > DEBUG > INFO > WARN > ERROR)
export TF_LOG=DEBUG
terraform plan

# Write logs to a file
export TF_LOG_PATH=./terraform-debug.log

# Use the console to test expressions
terraform console
# > var.environment
# > aws_instance.app[0].public_ip
🔍
When in doubt, read the plan carefully

terraform plan tells you exactly what will happen. Look for -/+ (destroy and recreate) symbols, which indicate a forced replacement — often surprising. The # forces replacement annotation shows the culprit attribute.
